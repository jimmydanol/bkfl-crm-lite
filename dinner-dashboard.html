Exact Thursday developer-team ask

The team is not being asked to redesign bankruptcy practice management or start coding from screenshots. The assignment is to turn the two behavioral prototypes and tonight’s approved rules into one buildable, secure plan for the debtor-to-Stage-4 vertical slice.

“Please return one recommended implementation path—not three unranked options—for moving a debtor from New Lead through Intake Completion, Document Review, and Attorney Review to Ready for Petition Prep. Map the prototypes onto the existing Azure platform, expose contradictions before estimating, and show us what a secure staging milestone requires.”

Required return package

Ask for all eight. Every artifact should label confirmed rules, developer assumptions, and decisions still owed by Matt.

01
Architecture + ownership map

Show existing Azure services, proposed Intake and CRM boundaries, integrations, tenant boundary, and the system of record for Contact, Lead, Intake Package, Document, Task, Communication, lifecycle Stage, and Matter.

02
Data + event model

Define entity IDs, immutable versus mutable records, gate state, audit history, idempotency, and the events that advance or regress Intake Completion, Document Review, Attorney Review, and Stage 4.

03
API + shared-contract map

Specify Intake submission, the locked Intake Package, doc_type and files[] bundles, skip/ruling codes, Client Portal token flow, uploads/reasons, corrections, and versioned AI output contracts.

04
Security + operations assumptions

State identity and roles, tenant isolation, encryption, private Blob access, malware/type/size controls, token expiry/revocation, audit logging, retention/deletion, secrets, AI-provider boundaries, and dev/staging/production separation.

05
Milestones + acceptance criteria

Break the slice into demoable vertical milestones. For each: inputs, observable behavior, owner, test data, definition of done, dependencies, and the exact acceptance walkthrough against Matt’s prototypes.

06
Estimate + dependency register

Give ranges tied to staffing and assumptions—not false precision. Name third-party services, accounts/keys, migration needs, unresolved decisions, technical spikes, risks, and what would materially change the estimate.

07
Staging-build proposal

Recommend the first deployable staging slice, environment owner, seeded demo scenarios, observability, rollback approach, review cadence, and the date when Matt can test an end-to-end happy path plus reopening cases.

08
Contradiction + decision log

List every rule conflict with source links, impact, recommended resolution, decision owner, and deadline. Do not bury product or legal questions inside architecture assumptions or estimates.

What a useful answer must make explicit

No estimate or build commitment should hide an unresolved legal/product rule.

Answer before estimating

  • Extend the existing Azure platform, build the canonical BK FastLane stack, or integrate both—and why.
  • Which system owns each record and event, including Lead-to-Matter conversion.
  • Which facts staff can waive, which warnings block, and which rulings require an attorney.
  • Whether counseling blocks Stage 4 or filing only.
  • Canonical Client Portal route, token scope, expiry, identity check, and revocation.
  • Exactly what reopens each completed gate and how Stage 4 regresses.

Definition of a credible response

  • One recommended path with rejected alternatives and tradeoffs.
  • Source links for every interpreted product rule.
  • Assumptions separated from approved decisions.
  • Estimate ranges connected to scope, staffing, dependencies, and risks.
  • Acceptance tests include happy path, missing facts, rejected documents, corrections, late uploads, and stage regression.
  • Staging uses synthetic cases only and never treats GitHub Pages as the security model.
